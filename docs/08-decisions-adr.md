# 08. 決定ログ / ADR

確定済みの設計判断と根拠。**Claude Code はこれらを再議しないこと。** 真に見直しが必要なら根拠を添えて明示的に提起する（黙って逸脱しない）。

形式: 文脈 / 決定 / 根拠 / 帰結。

---

## ADR-01 データソース = yfinance
- **文脈:** スイング判断に十分な日足/週足/月足が必要。
- **決定:** yfinance（Yahoo Finance）を採用。
- **根拠:** 日足/週足/月足は履歴ほぼ無制限。日本株/米株/主要クリプトを単一経路で取得可。
- **帰結:** 日中足は履歴制限（1m=7日、〜90m=60日、1h=730日）。v1 はスイング限定で日中足不使用。

## ADR-02 取得方式 = Python サイドカー
- **文脈:** Yahoo は公式 API 廃止、cookie/crumb 認証・調整・price repair が必要。
- **決定:** yfinance ラッパを Tauri サイドカー（PyInstaller/Nuitka）として同梱。Rust が起動・JSON/Parquet 受領。
- **根拠:** 認証/調整の Rust 再実装は割に合わない。取得（Python）と計算（Rust）の分離で「計算の単一真実源」は保たれる。
- **帰結:** バンドルサイズ増・OS別ビルド。CI（GitHub Actions）でサイドカーをビルド/署名。
- **代替案（不採用）:** Rust HTTP 直叩き。Python 不要だが認証保守コスト。

## ADR-03 タイムフレーム = 日足（主軸）/ 週足 / 月足
- **決定:** 1d を主軸、1wk・1mo を MTF（上位足文脈）に。
- **根拠:** いずれも履歴無制限で MTF が安定。スイングの保有期間（約1–4週）は週足スケール。
- **帰結:** 近接度（タイミング）は日足から読む。週足/月足は方向ゲート/修正子。

## ADR-04 更新方式 = EOD バッチ + オンデマンド（リアルタイム不要）
- **決定:** リアルタイム/ストリーミングは v1 スコープ外。
- **根拠:** 主用途はスイングのエントリー判断。EOD で十分。
- **帰結:** WebSocket 層なし。キャッシュ + 差分更新で運用。

## ADR-05 暗号資産 = Yahoo 銘柄（BTC-USD 等、日足/週足）
- **決定:** クリプトは yfinance の Yahoo 銘柄で扱う。
- **根拠:** スイングなら日足/週足で実用十分。単一データ経路を維持。
- **帰結:** Binance 等の取引所ネイティブ・3D 足は不可。必要なら将来プロバイダ追加（拡張点として確保）。

## ADR-06 計算の単一真実源 = Rust
- **決定:** 全インジケーター/レジーム/スコア/近接度を Rust 純粋関数で。フロント・チャートは再計算しない。
- **根拠:** チャートとスコアの不一致を構造的に防ぐ。`rayon` で銘柄横断並列。
- **帰結:** フロントは Rust 由来の系列/スコアを描画するだけ。

## ADR-07 レジームゲーティング（逆張り符号反転）
- **決定:** スコアリングは regime 判定を起点に、トレンド中は逆張りサブスコアを符号反転/ゼロ化してから合成。
- **根拠:** 逆張りと順張りは正反対のシグナル。単純加算は自己相殺する。トレンド中のオーバーソールドは継続買い。Connors RSI2 が200日MAフィルター下でのみ機能する事実とも整合。
- **帰結:** `scoring/weights.rs` にレジーム別重み表 + 符号規則。チューニング対象（P8）。

## ADR-08 二軸評価（方向スコア + 近接度スコア）
- **決定:** 方向/確信度（−100..+100）と エントリー近接度（0..100）を分離。
- **根拠:** 方向が強くても「もう動いた後」がある。欲しいのは「これからトリガー/いまトリガー」。
- **帰結:** `SymbolScore` に両軸 + `SignalState`（Primed/Triggered/Active）+ `actionability`。リストは actionability 降順。

## ADR-09 MTF = α 加重 + 週足ゲート（強）+ 月足ソフト修正子
- **決定:** `Score_MTF = 0.55*日足 + 0.30*週足 + 0.15*月足`。weekly_gate = 1.0/0.8/0.4。monthly_mod = 1.1/0.85（**ハードゲート禁止**、無効化可）。
- **根拠:**
  - 日足過半: タイミングは日足にしかない。上位足に薄めない。
  - 週足=月足の2倍: 保有期間≒週足スケール。幾何級数的減衰とも整合。
  - 月足ソフト: 月足は遅く、ハードゲート化すると大底で日足/週足が反転した初動を長期間ブロックする（任天堂チャートの局面）。
  - 週足は強ゲート: 上位足は"票"でなく"文脈"。逆行は減衰（200MA/上位MACDフィルターが支持）。
- **帰結:** α とゲートで週足は二重計上気味だが捉える情報が違う（強さ vs 方向整合）。エッジはゲート側、α は補助。定数は P8 で調整、不可侵でない。

## ADR-10 重み・閾値 = ユーザー調整可（プリセット付き）
- **決定:** MTF・ゲート・閾値・近接度パラメータを `Config`/`MtfConfig`/`ScanConfig` で調整可能。プリセット Conservative/Standard/Aggressive。
- **根拠:** 市場/資産で最適値が異なる。再現性のためスキャン時に設定をスナップショット（`scan_runs.config_json`）。
- **帰結:** マジックナンバーをコードに埋めない。

## ADR-11 「Triggered」鮮度 `fresh_bars_n` = 既定 1–2、設定可
- **決定:** 発火後 1–2 バー以内を「新鮮（いま出番）」とする。設定で変更可。
- **根拠:** スイングで「いま仕掛けられる」を狭く定義。出遅れ（Active）は減点して沈める。
- **帰結:** 状態機械と鮮度減衰に反映。

## ADR-12 クライアント提案連携 = v1 非統合（将来拡張）
- **決定:** Alpha Valuation Terminal の提案フローへの統合は v1 では行わない。
- **根拠:** スコープを絞る。出力（CSV/JSON エクスポート）を疎結合に保てば後から統合可能。
- **帰結:** エクスポート形式を将来連携を見据えて素直に保つ。

## ADR-13 ゴールデン値基準 = TA-Lib/pandas-ta 数値 + TradingView 目視
- **決定:** 単体テストの数値基準は TA-Lib/pandas-ta、視覚照合は TradingView。
- **根拠:** TradingView/Pine の内部計算はバージョン依存でビット再現が難しい。TA-Lib は決定的で文書化済み。ただし RSI/ADX seeding 等で微差があるため目視で流儀差がシグナルに効かないことを確認。
- **帰結:** 境界に効く差は TradingView 流儀へ寄せる（本ツールは TradingView 由来指標の再現が意図）。

## ADR-14 チャートマーカー = Supertrend 脚 × スコア閾値の合流（閾値クロス単独から改訂）
- **決定:** チャートの BUY/SELL マーカーは「Supertrend の方向脚ごとに最大1個、脚の方向をスコアが初めて閾値強度（buy/sell_threshold）で確認したバー」に表示。スコアが既に閾値超えの状態で脚が転換した場合はフリップバーに表示。スキャナー側の Triggered 判定（閾値クロス、docs/03）は変更しない。
- **根拠:** 当初の FR-8（`Score_final` 閾値クロスをそのまま描画）はスコアの閾値付近の振動で密集・逆行マーカーを量産し、エントリーポイントとして視認不能だった。キャッシュ実データ（9銘柄・平均6,862バー）で候補規則を比較検証:

  | 規則 | マーカー数/銘柄 | hit10 | PF10 |
  |---|---|---|---|
  | 閾値クロス（旧） | 232.8 | 55.6% | 1.25 |
  | Supertrend フリップ素通し | 236.3 | 50.3% | 1.01 |
  | 脚×スコア合流（採用） | 116.6 | **56.0%** | 1.16 |

  生フリップはエッジ皆無（トレーリングストップの転換は既に3ATR動いた後）。閾値クロスの高PFは同一の動きを重複カウントした密集分によるもの。合流規則は的中率最高・1脚=1マーカーで重複なし・チャートに描画済みの Supertrend 線と視覚的に整合（TradingView の Adaptive Trend Sentinel の Long/Short ラベル + Q-Trend のトリガーの2層構成に相当）。診断は `src-tauri/tests/marker_experiment.rs`（`--ignored`、`ALPHA_RADAR_DB` 指定で再実行可）。
- **帰結:** マーカーは表示規則であり近接度エンジン・評価ハーネスとは独立。閾値は従来通り設定・プリセットに追従。規則本体は `scoring::marker_events` に一元化し、チャート描画とランキングの「的中率」列（`marker_hit_rate`、`marker_horizon_bars` 営業日後の順行率バックテスト）が常に同一マーカーを参照する。

## ADR-15 Q-Trend 表示レイヤー（ADR-14 マーカーと併存）+ 前兆検出
- **決定:** TradingView「Q-Trend」（tarasenko_、close/200/ATR14/mult1.0/Type A）を Rust に忠実移植し（`indicators::trend::qtrend`）、チャートに **既存 ADR-14 マーカーと併存する独立レイヤー**として表示する: ラチェット式トレンドライン（青）+ フリップマーカー（QT BUY 青 / QT SELL 橙 / STRONG 修飾）+ **前兆サークル**（`QT前兆`、フリップ閾値まで `qtrend_precursor_atr`（既定 0.5）ATR 以内かつ終値が閾値方向へ動いた最初のバー、1レグ1回）。トグル「Q-Trend」「QT前兆」は既定 ON。
- **根拠:** ユーザー観察により Q-Trend のフリップタイミングの視認性が高く、「フリップ約1日前が最適な仕掛け」仮説が提起された。ADR-14 の合流マーカーは検証済みのため置き換えず、Q-Trend を表示レイヤーとして追加し目視比較を可能にする。前兆は「フリップ閾値までの距離（ATR 単位、`dist_flip_atr`＝先読みなし）」として定式化。
- **検証優先ポリシー（原則4）:** `marker_experiment.rs` にルール **H**（フリップ当日）/ **I**（フリップ1本前＝後知恵上限）/ **J**（前兆＝リアルタイム近似）を追加し実キャッシュで比較。**仮説支持 = I が H を ret10/PF で上回り、かつ J ≈ I**。近接度エンジン/ランキングへの Q-Trend 組み込みは、この検証で優位性が確認されてから**別 ADR** で決定する（本 ADR は表示のみ）。
- **帰結:** `scoring::marker_events`・`SymbolScore`・的中率列は不変更。スコア/レジーム/近接度への影響なし。ゴールデンテスト（p=50 主 / p=200 補助）付き。ウォームアップは最大1本の乖離を許容（docs/02 の規約）。
- **検証結果（実施済み・9銘柄・平均6,862バー）:**

  | 規則 | n/銘柄 | hit5 | hit10 | ret10 | PF10 |
  |---|---|---|---|---|---|
  | H: QTフリップ当日 | 604.6 | 50.6% | 50.2% | −0.02% | **0.99** |
  | I: フリップ1本前（後知恵） | 604.6 | **76.1%** | 67.6% | +3.13% | **3.18** |
  | J: 前兆（リアルタイム） | 362.9 | 48.6% | 49.5% | −0.26% | **0.91** |

  判定: **仮説は取引可能な形では不支持**。I の圧倒的成績は**ルックアヘッド**（フリップはそのバーの大きな値動きで定義されるため、1本前エントリーはその値動き自体を刈り取る後知恵）であり、リアルタイム近似の J（0.5ATR接近+同方向モメンタム）は H をも下回った（前兆→3本以内フリップの精度55%、カバレッジ35%、平均リード1.59本）。よって **Q-Trend/前兆のランキング組込みは行わない**（本 ADR のポリシー通り）。Q-Trend レイヤーは視認用の表示専用として維持。検証済みエントリー規則は引き続き ADR-14 の合流マーカー（hit10 56.0% / PF 1.16）。

## ADR-16 ST フリップ表示レイヤー + 直近マーカー列 + 動的ペイン/ウインドウ追従
- **決定1（ST フリップ）:** Supertrend の方向フリップに `ST LONG / ST SHORT` マーカーを表示するトグル「STフリップ」を追加。TradingView の Adaptive Trend Sentinel の Long/Short ラベルに相当する**視覚比較専用**レイヤー。ATS はクローズドソースのため正確な複製は不可能で、同型の ATR トレーリングストップ＝Supertrend で近似する。**単独エッジは実測でなし**（ADR-14 規則B: hit10 50.3% / PF10 1.01）— エントリー規則は引き続き ADR-14 の合流マーカー。
  - **追記（パラメータ分離）:** スコアリング用 Supertrend（10 / 3.0）を流用すると ATS（ステータス行 "2 10"）よりフリップが遅く・少なくなりタイミングがずれるため、表示専用の **`stflip_atr` / `stflip_mult`（既定 10 / 2.0）** を新設し別計算とした。スコア・確定マーカーへの影響なし。既定トグルはユーザー指定により **ON**（初期ビュー: EMAリボン/MACD/Squeeze/Q-Trend/QT前兆/STフリップ）。
- **決定2（直近マーカー列）:** レーダーに「直近マーカー」列を追加。確定（ADR-14）/ QTフリップ / QT前兆の**全種から最新1件**を種別×方向の配色バッジ＋経過（本日/N日前、営業日ベース）で表示し、経過日数でソート可（既定昇順=新しい順）。同一バーのタイブレークは 確定 > QTフリップ > 前兆。`SymbolScore.last_marker`（`{kind, dir, bars_ago}`）として算出・`scores` テーブルへ追加移行（ALTER TABLE 追加パターン）。
- **決定3（レイアウト）:** チャートはウインドウ追従（`.chart-page` 100vh フレックス + `.chart-host` flex:1、`autoSize` 追従）。ペイン高さは `setStretchFactor` で動的化 — MACD/Squeeze/スコアの OFF でペインを 2px まで収縮し価格ペインが残余を吸収（`setHeight` は最小30pxクランプ+兄弟再膨張のため**使用禁止**）。「スコア」トグル新設。`layout.panes.enableResize: false` でセパレータドラッグとの競合を防止。ADR-14 時代の「固定4段の割り切り」は本決定で置換（ズームは維持される — ペイン構造不変・factor 変更のみ）。

## ADR-17 UI 状態の永続化 = `config.json` とは別ファイル（`ui_prefs.json`・Rust 側は不透明保持）
- **決定:** チャートの表示トグル（`ChartVisibility` 10個）は `app_data_dir/ui_prefs.json` に保存し、次回起動時に復元する。Rust は `serde_json::Value` として**スキーマを解釈せず**読み書きする（`get_ui_prefs` / `update_ui_prefs`）。スキーマの所有はフロントエンド（`lib/ui-prefs.ts`）。**足種（tf）とレーダーのソート/フィルタは永続化しない**（セッション限り）。
- **根拠:** `ScanConfig` は毎スキャン `scan_runs.config_json` に**スナップショットされる再現性の監査記録**（ADR-10）であり、装飾的な表示フラグを混ぜると監査記録が汚れ、`/settings` の往復スキーマも変わる（かつ `Copy` 制約もある）。表示フラグは**計算ではない**ため ADR-06「Rust が計算の単一真実源」は及ばない。トグルは実績として3度増えており（ADR-15/16 で `score`/`qtPrecursor`/`stFlip`）、不透明保持なら追加時に Rust 変更が不要。`indicators`/`regime`/`weights` を `Record<string, unknown>` で素通しする既存の前例と同型。検索語を永続化しないのは、前回の絞り込みが残ってリストが空に見える事故を避けるため。
- **帰結:** 読み込みは寛容（**既知キーかつ boolean のみ採用**、不明キー無視、新トグルは既定、破損時は既定へ縮退、不在は `null`）。書き込み失敗は UI に出さず `console.warn` のみ。ハイドレーション中のユーザー操作は `DEFAULT_VISIBILITY` の**参照同一性**で判定して上書きしない（`useState({...DEFAULT_VISIBILITY})` に変えると壊れる）。**`ScanConfig` に UI 状態を足さないこと。**

## ADR-18 macOS 配布 = arm64 単体 `.dmg`・未署名（universal 化は不可能）
- **決定:** リリース CI は `macos-latest`（Apple Silicon）で **arm64 の `.dmg` を1本**生成する。`--target universal-apple-darwin` は採用しない。署名・公証は行わない。Windows は NSIS。Linux は対象外。
- **根拠:** **サイドカーを universal 化できない** — PyInstaller の `--target-arch universal2` は universal2 のインタプリタと全ネイティブ依存（pandas 等）の universal2 wheel を要求するが、uv 管理の python-build-standalone は単一アーキ（「Python は uv のみ」はプロジェクト規約）。また one-file バイナリ2本の `lipo` 統合も**不可**（PyInstaller が Mach-O イメージの外に追記する CArchive が失われ、起動できないバイナリになる）。よって Intel 対応は universal ではなく **`macos-13` ランナーによる2本目の dmg**（必要になれば matrix に1行追加）。Windows からのクロスビルドは Tauri（macOS SDK 必須）・PyInstaller の双方で原理的に不可能。
- **帰結:** Intel Mac は非対応。未署名のため Gatekeeper 回避が必要 — **特に同梱サイドカー（`fetch`）にも個別に隔離属性が付き、「このまま開く」だけではアプリは起動してもスキャンが必ず失敗する**ため、`xattr -dr com.apple.quarantine` を **`.app` ツリー全体**に適用する必要がある（README / `.github/release-notes.md` に明記）。正式対応は Apple Developer Program（$99/年）+ Developer ID 署名/公証。

## 拡張点（v1 スコープ外・設計上確保）
- 取引所ネイティブ暗号資産（Binance/Bitbank・非標準足）= データプロバイダのプラグイン追加。
- 日中足版（1h は730日取得可）= 短期トレード用プリセット。
- AVT 提案フロー統合。
- リアルタイム/ストリーミング監視。
