# Alpha Radar 要求定義書

**プロジェクト:** alpha-radar（製品名: Alpha Radar — Swing Entry Confluence Scanner）
**ポジショニング:** AILEAP テクニカル分析レイヤー。`alpha-sentinel` / Alpha Valuation Terminal（ファンダ）と `prediction-eval`（検証）の間を埋める、**スイングトレードのエントリー近接銘柄をスコア化するスキャナー兼チャートツール**。
**版:** v0.3（全決定確定・実装着手可）

---

## 0. 設計思想（最初に読むこと）

本ツールの目的は「精度の高いインジケーターを見つけて表示する」ことでは **ない**。実証データ上、単体で高エントリー精度を持つインジケーターは存在しない（外部の「勝率XX%」はSEO/過剰最適化が大半で信用しない）。本ツールが追求するのは以下。

1. **期待値（expectancy）** を改善する設計。勝率は二次指標。
2. エッジの源泉である **レジーム選別 × コンフルエンス × ATRリスク管理** を、再現可能なスコアとして定量化する。
3. 「**いま／もうすぐ仕掛けられる銘柄**」を近接度で前面に出す。方向だけでなくタイミングを定量化する。
4. スコアの妥当性を **自前で検証**（`prediction-eval` の二項検定 / MFE-MAE）してから信頼する。

> **最重要原則①：** 逆張り指標と順張り指標は正反対のシグナルを出す。レジーム判定なしに加算すると自己相殺する。スコアリングは必ず「レジーム判定 → カテゴリ重みの動的切替」を最初に行う。
>
> **最重要原則②：** 方向スコアが強くても「既に動いた後」のことがある。本当に欲しいのは「これからトリガーする／いまトリガーした」銘柄。ゆえに **方向スコア（-100〜+100）** と **近接度スコア（0〜100）** を分離する。近接度は**日足**から読む。

## 1. スコープと前提（確定）

### 1.1 データソースとタイムフレーム

**データソース: yfinance（Yahoo Finance）。** 履歴制限（インターバル依存）:

| インターバル | 取得可能履歴 | 扱い |
|---|---|---|
| 1m | 直近7日 | 対象外 |
| 2m–90m | 直近60日 | 対象外 |
| 60m/1h | 直近730日 | 対象外（将来の短期版で検討） |
| **1d（日足）** | ほぼ全期間 | **主軸** |
| **1wk（週足）** | ほぼ全期間 | **MTF: トレンド文脈** |
| **1mo（月足）** | ほぼ全期間 | **MTF: レジーム/マクロ文脈** |

スイング判断が目的のため日足を主軸、週足・月足を上位足ゲートに用いる（履歴無制限でMTFが安定）。日中足は60日/730日の壁があり、かつリアルタイム不要のため v1 対象外。

### 1.2 対象資産

| 資産クラス | 取得方法 | 備考 |
|---|---|---|
| 日本株（TSE） | yfinance（例 `7974.T`） | 主対象 |
| 米株 | yfinance（例 `AAPL`） | 同一経路 |
| 暗号資産 | yfinance の Yahoo 銘柄（例 `BTC-USD`） | **日足/週足のみ。取引所ネイティブ・3D足は不可（将来拡張）** |

### 1.3 更新方式・データ取得アーキテクチャ

- **更新方式:** リアルタイム不要。**EODバッチスキャン + オンデマンド更新**。
- **取得方式: Pythonサイドカー方式。** yfinance ラッパ（PyInstaller/Nuitka）を Tauri 2 サイドカーとして同梱。Rust が起動 → JSON/Parquet で受領 → SQLite キャッシュ。
  - 採用理由: yfinance は Yahoo の cookie/crumb 認証・splits/dividends 調整・price repair を裏で処理する事実上の標準。Rust 再実装は割に合わない。データ取得（Python）と計算（Rust）を分離するため「計算の単一真実源」は保たれる。
  - 留意: バンドルサイズ増・OS別ビルド。サイドカーのビルド/署名は CI（GitHub Actions）へ。
- yfinance は失敗が一定割合・レイテンシ数百ms〜。**キャッシュ・リトライ・指数バックオフ・一括取得**前提。

### 1.4 既存資産の再利用

- **SQLite + DuckDB**: OHLCV・スコアキャッシュ（`alpha-sentinel` 同方式）
- **`prediction-eval` の評価ロジック**（二項検定 / MFE / MAE）: スコアモデル検証に流用
- **lightweight-charts v5**（マルチペイン）

## 2. 機能要件

### FR-1. データ取込（CSVドラッグ&ドロップ）

- **UI:** ドラッグ&ドロップボックス。CSV ドロップで即スキャン。ファイル選択も併設。
- **CSV スキーマ（最小）:** `symbol`（必須, yfinance形式）, `name`（任意）, `asset_class`（任意, equity/crypto, 省略時推定）, `weight`（任意）。
- **動作:** 各 symbol を yfinance（サイドカー）で 日足/週足/月足 取得 → SQLite キャッシュ（差分更新）。取得失敗は**行単位でエラー収集**しスキャン継続、UI に集約表示。進捗（n/N）表示。

### FR-2. インジケーター計算エンジン

Rust の純粋関数。`rayon` で銘柄横断並列。各指標は「生値」と「正規化サブスコア `s_i ∈ [-1,+1]`」を返す。週足/月足は**各足を直接取得**して計算（リサンプリングのズレ回避）。詳細は `docs/02-indicators.md`。

- **トレンド系:** ADX/DMI(14), Supertrend(ATR10/3.0), EMAリボン(20/50/200), 一目均衡表(9/26/52)
- **モメンタム系:** MACD(12/26/9, MTF), RSI(14), Squeeze Momentum(BB20/2.0, KC20/1.5, linreg), TSI(25/13)
- **逆張り系:** Connors RSI(2)+200MAフィルター, Bollinger %B(20/2.0), Williams %R(14), MA乖離zスコア(SMA20/100本)
- **ボラ/フィルター:** ATR(14), BB幅/ケルトナー(20), Choppiness Index(14)

### FR-3. レジーム判定（スコアリングの前段）

最初に必ず実行。`regime ∈ {TREND_UP, TREND_DOWN, RANGE, TRANSITION}`。ADX>25=トレンド（方向は±DI）、ADX<20=レンジ、中間=TRANSITION。Choppiness で補強。レジーム別カテゴリ重みは `docs/03-scoring.md`。**トレンド中は逆張りサブスコアの符号を反転 or ゼロ化**。

### FR-4. 方向スコアリング（コンフルエンス + MTF）

```
S_f   = mean(s_i for i in f)              # カテゴリ平均
S_raw = Σ_f w_f(regime) * S_f             # レジーム重み（逆張りは符号調整済み）
S_tf  = S_raw * gate(volatility, liquidity)
Score_tf = clamp(round(S_tf*100), -100, +100)
```

MTF（日足/週足/月足、**確定仕様**）:
```
Score_MTF = 0.55*Score_daily + 0.30*Score_weekly + 0.15*Score_monthly
weekly_gate = 1.0(整合) / 0.8(中立) / 0.4(逆行)
monthly_mod = 1.1(整合) / 0.85(逆行)   # ★ハードゲートにしない
Score_final = clamp(Score_MTF * weekly_gate * monthly_mod, -100, +100)
```
近接度は日足から読む。詳細・根拠は `docs/03-scoring.md`。

### FR-5. エントリー近接度エンジン（Imminence）★主機能

方向とは別軸で「いま／もうすぐ仕掛けられる」度合いを 0〜100 で算出。

- **状態:** `Primed{Buy/Sell}`（形成中・未発火）/ `Triggered{Buy/Sell}`（直近 `fresh_bars_n` バー内・新鮮）/ `Active{Buy/Sell}`（発火済・進行中＝出遅れ）/ `Neutral`。
- **近接度コンポーネント（各 [0,1]）:** 合成しきい値への接近+速度(`p_thresh`)、スクイーズ近接(`p_sqz`)、Connors RSI(2)トリガー近接(`p_cr`)、主要MA/バンドへの接近(`p_pull`, ATR単位)。
- **アクション度（並べ替え用）:** `actionability = proximity_score * (0.5 + 0.5*|Score_final|/100)`。
- 数式・状態遷移・鮮度減衰は `docs/03-scoring.md`。

### FR-6. 近接度ランキングリスト（主画面）

`actionability` 降順表示。買い接近=緑系/売り接近=赤系で2分割、中立は折りたたみ。列: 銘柄/名称/状態バッジ(Primed/Triggered/Active)/近接度(色バー)/方向スコア/**的中率（FR-8 マーカー規則の日足バックテスト、`marker_horizon_bars` 営業日後の順行率 + 標本数）**/レジーム/カテゴリ内訳/発火経過バー/ATR/損切り目安/更新時刻。ソート・フィルタ・CSV/JSONエクスポート。行クリックでチャートへ。詳細 `docs/05-ui-spec.md`。

### FR-7. チャート表示（ドリルダウン）

lightweight-charts v5 マルチペイン。①価格(ローソク+EMAリボン/Supertrend/一目+買い売りマーカー) ②MACD(MTF, 4色ヒスト) ③Squeeze Momentum(linreg+スクイーズ点) ④合成スコア(しきい値線併記)。上部に MTF サマリー（日足/週足/月足のレジーム・velocity）。指標値は Rust 計算結果を受領（再計算しない）。`attributionLogo` 必須。

### FR-8. シグナル / マーカー生成

**スキャナーのシグナル**（Triggered 判定）は従来通り閾値クロス（`docs/03` の状態機械）:

```
Buy : Score_final が +閾値 を上抜け AND 上位足レジーム非反対
Sell: Score_final が −閾値 を下抜け AND 上位足レジーム非反対
```

**チャートの売買マーカー**は Supertrend 脚 × スコアの合流点（ADR-14 で改訂）:

```
Buy : Supertrend 上昇脚の中で、スコアが初めて +閾値 以上になったバー（脚ごとに最大1個）
Sell: Supertrend 下降脚の中で、スコアが初めて −閾値 以下になったバー（脚ごとに最大1個）
```

価格ペインに `BUY @ price` / `SELL @ price` マーカー（series-markers, `zOrder`）。閾値はユーザー調整可（プリセット: 保守/標準/積極）。単純な閾値クロス表示はスコアの振動で密集・逆行マーカーを生む（実データ検証で判明）ため、トレーリングストップ脚を方向軸・スコアをトリガーとする2層構成に変更。

### FR-9. バリデーション / 評価ハーネス

`prediction-eval` 流用。方向的中率の**二項有意検定**、MFE/MAE分布、シグナル後Nバーのリターン・期待値・プロフィットファクター、**近接度の妥当性**（高近接度エントリーがリターンで分離するか）。レジーム別・asset_class別に層別。**ウォークフォワード**（IS/OOS分離）。詳細 `docs/07-testing.md`。

## 3. 非機能要件

| 項目 | 要件 |
|---|---|
| 性能 | 数百銘柄のEODスキャンを Rust 並列で現実的時間内（キャッシュ前提）。yfinanceレート制限がボトルネック |
| 再現性 | 同一入力で同一スコア（浮動小数の決定性に留意） |
| キャッシュ | OHLCV・計算結果を SQLite 保存、差分更新 |
| レート制限 | yfinance呼び出しをスロットリング・指数バックオフ・失敗行リトライ |
| 拡張性 | 新インジケーター・新市場（取引所ネイティブ・日中足）をプラグイン的に追加可能 |
| ロギング | 取得失敗・計算異常を行単位で記録、UI 集約表示 |
